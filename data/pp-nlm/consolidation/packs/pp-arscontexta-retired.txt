description: Never spawn more than 6 agents or 8 background tasks simultaneously — hard cap to prevent freezing the machine

type: methodology

category: behavior

source: session-mining

session_source: c5fc7f99-1c39-4eb0-9cea-9a036e51dba9.jsonl

created: 2026-04-06

status: promoted

Enforce a hard cap of 6 simultaneous agents and 8 background tasks

What to Do

Before spawning any agent or background task, count how many are currently active. If the total would exceed 6 concurrent agents OR 8 total background tasks, hold off and wait for some to complete before launching more.

Scale the caps DOWN further based on work complexity:

Heavy computation (TS compile, test runs, large builds): cap at 4 agents / 6 tasks

Light research/read-only work: up to 6 agents / 8 tasks is acceptable

When uncertain, default to the lower bound

What to Avoid

Spawning 10, 12, or more agents at once because the work is theoretically parallelizable. Never prioritize theoretical throughput over hardware reality. Do not assume the machine can handle maximum concurrency just because the task list is long.

Why This Matters

Tyler's machine froze completely when 12+ agents were spawned simultaneously. This blocks all work and wastes more time than sequential execution would have. A frozen computer is strictly worse than a slower but stable one.

Direct quote: "Can't start 12 fucking agents at the same time... hard cap at 8 background tasks | 6 agents working simultaneously and/or parallel... and those hard caps need to be lowered depending on the amount of work being done and complexity."

Scope

Always. No exception unless Tyler explicitly gives the go-ahead for a higher count for a specific session.

What to Do

When computing component placement on a breadboard using 

BreadboardFit

 logic, always validate that physical dimensions and pin offsets strictly align with the standard 0.1-inch (2.54mm) grid matrix. Use bounding box collision detection and snap-to-grid constraints.

What to Avoid

Do not place components based purely on arbitrary pixel coordinates without snapping to the breadboard hole matrix, and do not ignore physical overlaps.

Methodology

How this knowledge system works, in ProtoPulse terms.

Principles

Prose-as-title.

 Note titles are complete claims or insights, not topic labels. "Decoupling capacitors should be placed within 5mm of IC power pins" beats "Decoupling Capacitors." The title IS the knowledge -- the body provides evidence and context.

Wiki links as graph edges.

 Every 

[[link]]

 is a deliberate connection between ideas. Links aren't decoration -- they're the structure that makes knowledge findable and composable. When you link two notes, you're saying "these ideas inform each other."

Topic maps as attention managers.

 Topic maps (MOCs) aren't folders -- they're curated views into the graph. A topic map for 

[[pcb-layout]]

 surfaces the notes that matter for PCB work without hiding anything. They're how you navigate 500 notes without drowning.

Capture fast, process slow.

 Drop raw material in 

inbox/

 immediately. Don't try to write perfect notes in the moment. The extraction pipeline exists precisely so you can be sloppy during capture and rigorous during processing.

The Processing Cycle

Extract

Take raw source material from 

inbox/

 and pull out individual knowledge notes. Each note gets one atomic idea with proper frontmatter (type, confidence, topics). This is where you transform someone else's words into your own understanding. Verbatim quotes are for source captures, not knowledge notes.

Connect

Find relationships between notes. Update topic maps. Add 

Relevant Notes

 links. This is where isolated facts become a knowledge graph. The question to ask: "What does this note change about how I understand the notes around it?"

Revisit

The backward pass. Go back to older notes and update them with new connections, new evidence, or new confidence levels. A note written three months ago might need 

superseded_by

 or might gain a connection to something just discovered. This is how knowledge stays alive instead of rotting.

Verify

Challenge claims against evidence. Does this still hold? Has the datasheet been updated? Did the implementation prove or disprove the theory? Update 

confidence

 fields. Mark 

outdated

 when appropriate. Link to the note that supersedes.

In Practice

The cycle isn't linear. You might extract three notes from a datasheet, immediately connect two of them to existing topic maps, and flag one for verification because it contradicts an existing claim. That's fine. The phases are thinking modes, not a rigid pipeline.

Condition-based maintenance kicks in when the vault gets unhealthy -- too many orphans, too many unprocessed captures, too many stale notes. The hooks and health checks surface these signals so you can respond before drift compounds.

 Topic maps (MOCs) aren't folders -- they're curated views into the graph. A topic map for 

[[pcb-layout]]

 surfaces the notes that matter for PCB work without hiding anything. They're how you navigate 500 notes without drowning.

Capture fast, process slow.

 Drop raw material in 

inbox/

 immediately. Don't try to write perfect notes in the moment. The extraction pipeline exists precisely so you can be sloppy during capture and rigorous during processing.

The Processing Cycle

Extract

Take raw source material from 

inbox/

 and pull out individual knowledge notes. Each note gets one atomic idea with proper frontmatter (type, confidence, topics). This is where you transform someone else's words into your own understanding. Verbatim quotes are for source captures, not knowledge notes.

Connect

Find relationships between notes. Update topic maps. Add 

Relevant Notes

 links. This is where isolated facts become a knowledge graph. The question to ask: "What does this note change about how I understand the notes around it?"

Revisit

The backward pass. Go back to older notes and update them with new connections, new evidence, or new confidence levels. A note written three months ago might need 

superseded_by

 or might gain a connection to something just discovered. This is how knowledge stays alive instead of rotting.

Verify

Challenge claims against evidence. Does this still hold? Has the datasheet been updated? Did the implementation prove or disprove the theory? Update 

confidence

 fields. Mark 

outdated

 when appropriate. Link to the note that supersedes.

In Practice

The cycle isn't linear. You might extract three notes from a datasheet, immediately connect two of them to existing topic maps, and flag one for verification because it contradicts an existing claim. That's fine. The phases are thinking modes, not a rigid pipeline.

Condition-based maintenance kicks in when the vault gets unhealthy -- too many orphans, too many unprocessed captures, too many stale notes. The hooks and health checks surface these signals so you can respond before drift compounds.

created: 2026-04-11

status: active

Recover user-side session messages from history jsonl when the full transcript was deleted by Claude Code retention

What to Do

When a session transcript at 

~/.claude/projects/<slug>/<uuid>.jsonl

 has been deleted by Claude Code retention cleanup (the typical window appears to be ~2-3 days of inactivity for older sessions), the conversation is NOT fully unrecoverable. 

~/.claude/history.jsonl

 stores every user prompt indexed by 

sessionId

. It can be used as a partial recovery vector for friction mining.

The working pattern:

# 1. Extract all user prompts for a target session
grep '"sessionId":"<uuid>"' ~/.claude/history.jsonl | \
  jq -r '.display' > /tmp/recovered-<uuid>.txt

# 2. Scan for friction signals (mirrors ops/queries/mine-session.sh taxonomy)
grep -iE "(^|[^a-z])(no,?|wrong|stop|don't|never|incorrect|actually|fuck|shit|absolutely|mandatory|must)([^a-z]|$)" /tmp/recovered-<uuid>.txt

# 3. Review hits semantically and write observation/methodology notes for
#    genuine friction patterns. Discard false positives.


bash

Important nuances:

Use grep, not streaming jq.

 

jq -r 'select(.sessionId == "...")' ~/.claude/history.jsonl

 silently returns zero results when the file contains any malformed JSON lines (observed: 119 parse errors in a 9664-line file). Pre-filter with 

grep '"sessionId":"<uuid>"'

 to get only the candidate lines, then pipe through 

jq -r '.display'

 for clean extraction.

Recovery is partial, not total.

 You get the user half of the conversation — every message typed or pasted, with pastedContents inlined. You do NOT get assistant responses, tool calls, tool results, or inline file content that Claude read. For friction mining this is usually enough because most friction signals (corrections, redirections, escalations, directives) originate on the user side.

Message counts vary wildly per session.

 In the 2026-04-11 Bucket E recovery pass: 10 target UUIDs yielded counts of 135, 172, 30, 0, 6, 0, 78, 63, 1, 1. Two sessions had zero recoverable messages (3aa5de23, 4335d89e) — those UUIDs apparently never had corresponding history.jsonl entries, possibly because they were very short or errored out.

The 

.project

 field in history.jsonl narrows by project

 when 

sessionId

 alone isn't specific enough across different repos.

What to Avoid

Do not assume transcripts are permanently lost

 after Claude Code retention cleanup runs. Always check history.jsonl first.

Do not restore recovered files to 

~/.claude/projects/<slug>/

 — they are not full transcripts and would confuse Claude Code's session management. Work with them in 

/tmp/

 only.

Do not fall back to destructive filesystem recovery

 (

extundelete

, 

photorec

) until history.jsonl has been checked. It's a much simpler and higher-success path.

Do not rely on Timeshift snapshots

 for transcript recovery on systems where 

/home/<user>/**

 is in the exclude list. On Linux Mint's Timeshift default config, ALL of 

/home/wtyler/**

 is excluded (confirmed via 

cat /timeshift/snapshots/<date>/exclude.list

). Timeshift backs up system files, not user data.

Do not stream-parse the entire history.jsonl

 with 

jq

 expecting filters to work. The file has corruption from partial writes or malformed entries; streaming jq queries return zero matches when any line parse fails. Grep-then-jq is the reliable pattern.

Why This Matters

Observed during the session-mining pipeline rebuild (

docs/plans/2026-04-11-session-mining-pipeline-rebuild.md

) while auditing the 10 "Bucket E" stubs — old-format session captures from March 2026 with valid UUIDs whose transcripts had already been deleted by retention. Initial analysis assumed they were unrecoverable ("the transcripts are gone"). A deeper dig revealed that 

~/.claude/history.jsonl

 contained 405 entries matching those 10 UUIDs, yielding 486 recovered user messages across 8 of the 10 sessions.

The recovered messages contained strong friction signals that validated three existing memory-space user-preference files:

memory/feedback_no_mediocrity.md

 — traced to session c5fc7f99 (2026-03-23), direct quote recovered: "Never do just enough to be able to say something is done. NOTHING IS DONE UNTIL THERE IS NOTHING LEFT THAT CAN BE DONE!"

memory/feedback_autonomous_waves.md

 — traced to session c5fc7f99 and c3aee506, with explicit escalation about unauthorized stopping

memory/feedback_no_stepping_on_teammates.md

 — traced to session c3aee506, direct quote: "the validation-decomp agent is working on shit... what are you ding?" (repeated 7 times escalating to profanity)

memory/enforce-hard-cap-on-concurrent-agents.md

 (and its ops/methodology sibling) — traced to session c5fc7f99, direct quote: "Can't start 12 fucking agents at the same time... hard cap at 8 background tasks | 6 agents"

These recoveries prove the memory system captured real operational learning. They also prove history.jsonl is a durable recovery vector that should be part of the standard session-mining pipeline going forward.

Cost of NOT using this technique: the 10 Bucket E stubs would have been terminally marked 

transcript-unavailable

 and 486 messages of user-side friction would have stayed unmined forever.

Scope

Applies ALWAYS when:

A session transcript at 

~/.claude/projects/<slug>/<uuid>.jsonl

 is missing or has been deleted, AND

The session happened on this machine (history.jsonl is per-user, not synced), AND

The goal is friction mining OR user-prompt archaeology (recovering what was asked, not what was done).

Does NOT apply when:

Full transcript reconstruction is needed (assistant responses, tool calls). history.jsonl does not contain those.

The session happened on a different machine. History.jsonl is local.

The session is within retention and the transcript still exists. Read the .jsonl directly.

The file 

~/.claude/history.jsonl

 is itself missing, corrupt past repair, or has been rotated out.

The recovery is a supplement to the normal mining pipeline, not a replacement. The 

ops/queries/mine-session.sh

 runner reads real 

.jsonl

 transcripts for full-fidelity mining; this recovery pattern is for when that file no longer exists.

Use /agent-teams for parallel implementation work, not raw parallel subagents

What to Do

When parallel execution is needed for implementation tasks (writing code, modifying files, running builds), invoke 

/agent-teams

 with the teammate configuration. Let the team system coordinate agents via the shared task list, Shift+Up/Down navigation, and direct teammate messaging.

) are acceptable.

What to Avoid

/agent-teams

. This includes using the Task tool with 

subagent_type: general-purpose

 and 

run_in_background: true

 for anything that writes files or modifies state.

/agent-teams

 to "save time" by launching bare parallel agents. Tyler sees this as the same violation regardless of the framing.

Why This Matters

Tyler made this explicit and emphatic: "FOR NOW ON, /agent-teams shall be used unless oked by me to not do so when multiple agents are executed simultaneously... no more of that parallel shit! do you fucking hear me?"

The agent-teams system provides coordination, shared task visibility (Shift+Up/Down), file ownership enforcement, and teammate messaging. Raw parallel subagents bypass all of this, leading to file conflicts and uncoordinated work.

Scope

All parallel implementation work. If Tyler explicitly says "go ahead and run them in parallel without teams" for a specific task, that overrides this rule for that task only.

description: When Bash permission-denies a destructive command like rm -rf, immediately switch to mcp__desktop-commander__start_process — do NOT hand off to the user

type: methodology

category: behavior

source: session-mining

session_source: 0364162b-c4cd-499d-a08e-5d700252e6bb

created: 2026-04-11

status: active

Switch to Desktop Commander immediately when Bash denies a destructive command never hand off manual steps to the user

What to Do

When 

Bash

 returns 

Permission to use Bash with command ... has been denied

 (or any variant of permission denial), and the command is a filesystem operation (

rm

, 

rm -rf

, 

mv

, bulk delete, sensitive read), 

immediately switch to 

mcp__desktop-commander__start_process

 with the same command. Complete the operation in that turn. Do not return a partial result and ask the user to run the commands manually.

Concretely: if the original 

Bash

 call was

Bash(command="rm -rf /home/wtyler/Projects/ProtoPulse/node_modules/.vite")


and it was denied, the next call in the same turn should be

mcp__desktop-commander__start_process(command="rm -rf /home/wtyler/Projects/ProtoPulse/node_modules/.vite")


If Desktop Commander is also unavailable (tool not loaded, server down), load it via ToolSearch or escalate to the user with a precise, one-sentence reason — not a list of shell commands for the user to run.

What to Avoid

Do not do any of these:

Give the user three 

rm -rf

 commands and say "run these, then I'll verify."

 This is a handoff, not a solution. Tyler's global CLAUDE.md explicitly says "Claude Code's permission system blocks access to sensitive files. When Bash or Read tools fail with permission errors, USE DESKTOP COMMANDER INSTEAD."

Try the same 

Bash

 command again

 hoping the permission changes. It won't.

Skip the operation and proceed to the next task.

 The half-finished state becomes a landmine the next session has to find and clean up.

Ask the user to grant permission or add the command to the allowlist.

 Unless the user has said "allow this for the rest of the session," permission dialogs are not the escape hatch — Desktop Commander is.

Look for an alternative tool other than Desktop Commander

 (e.g., 

find ... -delete

, 

node -e "fs.rmSync(...)"

) when Desktop Commander is the documented correct path.

Why This Matters

Observed directly in session 

0364162b-c4cd-499d-a08e-5d700252e6bb

 (2026-04-10 14:22-14:37 UTC). The agent was asked to fix a stale 

gemini-cli

 installation at 

~/.npm-global/

. The fix required 

rm -rf

 on several paths. 

Bash

 denied the commands. The agent's response, quoted from the stop-hook feedback:

"The assistant left the task half-done with manual commands for the user to execute rather than completing the cleanup and verification."

And later in the same session:

"The transcript shows the assistant encountered a permission error ('Permission to use Bash with command has been denied'), attempted to find an alternative tool (desktop-commander), failed, then handed off three manual steps to the user to 'run now' with a promise to 'verify the result for you after.'"

Tyler's global 

CLAUDE.md

 already contains the rule with explicit mapping table (

rm -rf dir/

 → 

mcp__desktop-commander__start_process

). The rule was violated anyway. This methodology note exists as a session-grounded reinforcement so the pattern isn't just prose in a config file — it's a captured learning with a specific failure referenced.

The cost of the failure: Tyler had to do the cleanup manually, and the gemini-cli version problem persisted across multiple sessions because each attempted fix ended in a handoff rather than a completion.

Scope

Applies ALWAYS when:

A 

Bash

 invocation returns a permission-denied message, AND

The command is destructive (

rm

, 

rm -rf

, 

mv

, bulk operations), OR reads/writes a sensitive file (

.env

, 

.mcp.json

, credentials, 

*.pem

, 

*.key

).

Does NOT apply when:

The denial is for a command that would be inappropriate regardless of tool (

rm -rf /

, mass-destructive operations on shared state). Stop and escalate instead of routing around.

The denial is for reading a file the user has said you shouldn't read (the file-guard hook exists for a reason — Desktop Commander is the escape hatch for permission-boundary ambiguity, not for ignoring explicit user restrictions).

Desktop Commander itself denied access or is unavailable. In that case, explain the situation in one sentence and wait for user direction.

The point is speed and completion: if you know the right answer (Desktop Commander) and it's documented in 

CLAUDE.md

, just use it. Don't deliberate, don't explain, don't hand off. Switch tools and finish the work.
