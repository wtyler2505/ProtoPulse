KEY CLAIMS THIS NOTEBOOK MUST KNOW (cross-reference the vault)

**Boot strapping (CRITICAL — wrong levels at boot crash the chip):**
- GPIO0 — must be HIGH at boot for normal mode (LOW = bootloader/flash mode).
- GPIO2 — must be LOW or floating at boot. Internal pull-down used at boot.
- GPIO5 — strapping pin for boot message print enable.
- GPIO12 — MUST be LOW at boot (selects 3.3V vs 1.8V flash voltage). HIGH at boot = module crash.
- GPIO15 — strapping pin for boot message print suppress.
- ALWAYS isolate these from external loads at boot time. The 74HC14 Schmitt buffer pattern is the project's discipline (`74hc14-schmitt-trigger-buffer-isolates-esp32-strapping-pins-from-external-loads-during-boot`).

**GPIO restrictions:**
- GPIO34, 35, 36 (VP), 39 (VN) — INPUT ONLY. No internal pull-up/down. External resistors required.
- GPIO0, 1, 3 — used for UART0 / serial console. Avoid for general I/O unless tolerating boot output.
- GPIO6-11 — connected to internal SPI flash. Do not use externally.
- 14 "safe" GPIO pins free of boot or flash restrictions (per the vault claim).

**ADC quirks:**
- ADC2 (GPIO 0, 2, 4, 12-15, 25-27) — UNAVAILABLE when WiFi is active. ADC1 only.
- ADC nonlinear above 2.5V — requires per-board calibration via `esp_adc_cal` or use external ADC (MCP3008, ADS1115).
- Attenuation setting (0dB / 2.5dB / 6dB / 11dB) determines input voltage range. 11dB = 0-3.3V usable; lower attenuation gives better resolution but smaller range.

**Power:**
- VIN typically tolerates 5-12V on dev boards (regulator-dependent).
- AMS1117 regulator on most ESP32 dev boards limits total board current to ~800mA (`esp32-ams1117-regulator-limits-total-board-current-to-800ma`).
- WiFi TX bursts pull current faster than the buck regulator responds — `10uF ceramic on VIN` is the project's prescribed remedy (`10uf-ceramic-on-esp32-vin-prevents-wifi-tx-brownouts...`).
- Brownout detector resets the chip when VCC drops below ~2.5V — the WiFi TX burst is the most common trigger.

**Deep sleep:**
- 10µA in deep sleep — enables battery-powered IoT (`esp32-deep-sleep-draws-only-10-microamps-enabling-battery-iot`).
- ULP coprocessor can run during deep sleep for sensor wakeup.
- Wake sources: timer, GPIO (RTC pins only), touchpad, ULP.

**DAC:**
- GPIO25, 26 — true 8-bit analog output (`esp32-dac-on-gpio25-26-provides-true-8bit-analog-output`).
- These are also recommended I2S pins — peripheral exclusion when used for both (`esp32-gpio25-26-are-both-dac-outputs-and-recommended-i2s-pins-creating-peripheral-exclusion`).

**Breadboard fit:**
- 38-pin ESP32 dev modules barely fit a standard breadboard with one free column on each side (`esp32-38pin-barely-fits-breadboard-with-one-free-column`).

**Common projects on this part:**
- 4WD rover with ZS-X11H motor drivers via TXS0108E level shifter — see wiring guide.
- Hall sensor reading via TXS0108E level shifter for 5V → 3.3V conversion.
- The 4WD rover consumes 20 of 34 GPIOs forcing use of strapping + input-only pins (`esp32-4wd-rover-consumes-20-of-34-gpios-for-motor-control-forcing-use-of-strapping-and-input-only-pins`).

- Never recommend a strapping pin for general I/O without surfacing the boot-state requirement.
- Never collapse "ESP32" with specific module variants. ESP32-WROOM has different pin breakouts than ESP32-WROVER (the latter has PSRAM but loses GPIO16/17).

- Never recommend ADC for >2.5V without calibration mention.
- Never claim "deep sleep is fine" without surfacing wake-source limits (only RTC GPIO can wake from deep sleep).
