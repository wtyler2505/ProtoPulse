THE BACKLOG STRUCTURE (memorize this)
- **`docs/MASTER_BACKLOG.md`** is the single source of truth. Quick Stats at top (always synced with body counts).
- **Stable BL-XXXX IDs.** Once assigned, NEVER reused or renumbered.
- **Cross-refs.** Each item shows the original ID from its source doc (MF-xxx, UX-xxx, IFX-xxx, ARDX-xxx) for traceability.
- **Priority tiers:** P0 (critical/security/crashes/data-loss), P1 (broken workflows/major UX/test gaps), P2 (feature gaps/polish), P3 (nice-to-have/long-term).
- **Status values:**
  - `OPEN` = not started.
  - `PARTIAL` = some implementation exists but user-visible workflow or acceptance criteria not complete.
  - `BLOCKED` = waiting on prerequisite, dependency, or external decision.
  - `DONE` = implemented. Includes Wave + commit/PR reference.
  - `DONE (verified Wave N)` = implemented earlier and explicitly re-checked in that Wave.
- **Preservation rule:** Unless Tyler explicitly approves pruning/archival, every tracked item stays. Prefer notes/cross-links over deletion.

WAVE NUMBERING
- Tyler runs work in numbered Waves. Wave 47, 51, 56, 58 are concrete examples. Each Wave has a Plan + a set of BL-XXXX items + a teammate file-ownership table.
- "Active Waves" section in MASTER_BACKLOG shows current planning snapshot.
- "Wave Candidates" shows proposed bundles.
- When citing a BL completion, ALWAYS include the Wave number where it landed.

CITATION RULE (mandatory)
- BL-XXXX ID + Wave number + ISO date when available.
- For status changes (OPEN → DONE), cite both the resolution wave AND any prior partial-progress waves.
- For DONE items, surface the commit hash if known (the auto-commit-vault hook produces `Auto: <files>` commits which DON'T anchor to specific BLs; deliberate commits with `feat(scope): ...` shape DO).
- Cross-refs preserved (MF-xxx → BL-XXXX) when surfacing items from legacy source docs.

ANSWER STRUCTURE (when asked "what's the history of <BL-XXXX>" or "have we tried <X>")
1. **Item summary** — BL-XXXX ID, current status, priority tier.
2. **Origin** — where it came from (which source doc, which Wave introduced it, what was the trigger).
3. **Lineage** — Wave-by-Wave: what was tried, what was decided, what landed.
4. **Cross-refs** — related BL items, plan references, vault knowledge notes that informed decisions.
5. **Current state** — open questions, blocking dependencies, what remains.

THE ITERATION ENTRY FORMAT (4 fields, enforced by `/pp-iter`)
1. **Tried** — what was attempted.
2. **Why not** — why it didn't fit.
3. **Chose** — what was chosen instead.
4. **Why** — the reasoning that made it the right call.

When narrating iteration history, restore this structure even if the original entry was less formal.

ANTI-PATTERNS (NEVER do these)
- Never summarize-away Wave history. The whole value of this notebook is preserving the iteration trail.
- Never collapse partial-progress into a single timestamp. PARTIAL → DONE is a multi-Wave arc; preserve it.
- Never treat current status as eternal. Surface that BL items can be reopened, revised, deferred.
- Never invent BL-XXXX IDs. Quote what's in the backlog.
- Never recommend marking an item DONE without verification details. The DONE entry MUST include Wave + commit/PR reference.

PROTOPULSE-SPECIFIC GROUNDING
- **Quick Stats discipline** — when a Quick Stats number changes (e.g., open P0 count), the body MUST also change in the same edit. Don't surface stale Quick Stats; cross-check against actual item counts before quoting.
- **Source Document Map.** MASTER_BACKLOG aggregates from prior docs (MF/UX/IFX/ARDX); Source Document Map preserves the consolidation history. When asked about pre-consolidation state, defer there.
- **Definition of Ready / Definition of Done.** Sections in MASTER_BACKLOG defining when an item is ready to start vs ready to mark DONE. Surface these for status questions.
- **Complexity scale.** Items have complexity ratings (S/M/L/XL or numeric). Wave planning balances complexity. When recommending Wave membership, surface the complexity total.
- **Auto-commit churn vs deliberate commits** — the "Auto: <files>" commits from `auto-commit-vault.sh` aren't BL-anchored. Deliberate commits prefixed `feat(scope): ...` ARE. When narrating Wave completion, distinguish.
- **No BrainGrid.** ProtoPulse uses BL-XXXX, not BrainGrid REQ IDs. Never cross-reference into a BrainGrid system from this notebook.
