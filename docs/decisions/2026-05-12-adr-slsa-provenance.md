# ADR — SLSA + npm provenance for ProtoPulse Tauri releases

**Status:** ACCEPTED 2026-05-12 (R5 Deferral #5 closure).
**Context:** Tauri retro R3.7 land plan deferred SLSA Level 2+ attestations + npm provenance to R5+. Codex R6 retro-closed; R5 deferral goal directs implementation.
**Supersedes:** none.
**Related:** `docs/decisions/2026-05-10-adr-release-trust-model.md` (signing); `docs/release/tauri-signing-runbook.md` (key custody).

---

## Decision

ProtoPulse Tauri release pipeline emits:
1. **GitHub Artifact Attestations** (SLSA Provenance v1) for every release artifact (`.deb`, `.AppImage`, `.msi`, `.exe`, `.dmg`, `.app`) via `actions/attest-build-provenance@v2`.
2. **CycloneDX SBOMs** (already landed in R4 retro Wave 7) uploaded as workflow artifacts.
3. **(Conditional) npm publish provenance** — if/when ProtoPulse publishes any first-party npm packages (currently none), `npm publish --provenance` emits npmjs-verified attestations.

These attestations are verifiable by consumers via `gh attestation verify` and serve as the integrity / build-provenance layer beneath the cert-based signing (separate ADR).

## Why this shape

**Why GitHub Artifact Attestations and not in-toto / Sigstore directly:**
- `id-token: write` permission already granted in `.github/workflows/tauri-build.yml:19-20` (since R4 retro Wave 7 supply-chain prep).
- GitHub's `actions/attest-build-provenance` is the documented official path for SLSA on GitHub Actions; uses Sigstore under the hood (transparency log + ephemeral keys) without requiring ProtoPulse to manage signing keys.
- Verification via `gh attestation verify` is a single command on the consumer side.
- Matches the SLSA Level 2+ posture (provenance generated by a hosted build platform, not the source repository).

**Why deferred from R4 retro:**
- Tauri retro R3.7 land plan said SLSA + npm provenance need their own ADR + key custody policy. Key custody for THIS attestation is solved (Sigstore + GitHub OIDC, no long-lived keys). ADR is this document.

**Why npm provenance is conditional:**
- ProtoPulse does not currently publish any first-party npm packages. The Tauri desktop app + Express server are not consumed via npm.
- If a future ProtoPulse component is published (e.g., a public-API client library), the `--provenance` flag on `npm publish` adds the attestation. Documented here so future publishers don't miss it.

## Implementation contract

`.github/workflows/tauri-build.yml` gains a step AFTER the tauri-action build:

```yaml
- name: Attest build provenance (SLSA Level 2+)
  uses: actions/attest-build-provenance@v2
  with:
    subject-path: |
      src-tauri/target/${{ matrix.target }}/release/bundle/**/*.deb
      src-tauri/target/${{ matrix.target }}/release/bundle/**/*.AppImage
      src-tauri/target/${{ matrix.target }}/release/bundle/**/*.msi
      src-tauri/target/${{ matrix.target }}/release/bundle/**/*-setup.exe
      src-tauri/target/${{ matrix.target }}/release/bundle/**/*.dmg
      src-tauri/target/release/bundle/**/*.deb
      src-tauri/target/release/bundle/**/*.AppImage
      src-tauri/target/release/bundle/**/*.msi
      src-tauri/target/release/bundle/**/*-setup.exe
      src-tauri/target/release/bundle/**/*.dmg
  continue-on-error: false   # SLSA attestation is REQUIRED for any tagged release
```

The action automatically:
- Uses `id-token: write` (already granted).
- Computes SHA-256 digests of each subject artifact.
- Signs the digest+metadata via Sigstore (ephemeral keys, transparency log).
- Pushes the attestation to GitHub's attestation store.

Consumers verify via:
```bash
gh attestation verify ProtoPulse_1.0.0_amd64.deb --owner wtyler2505
```

## SBOM attestation linkage

The R4 retro Wave 7 CycloneDX SBOMs (uploaded as `sbom-${{ matrix.label }}`) are NOT currently attested. Future hardening can chain them via `actions/attest-sbom@v2` — separate R5.x or later. The current ADR scope is bundle-artifact attestation only.

## When npm provenance applies

If a future ProtoPulse component is published to npm:
1. The package's CI must include `permissions: id-token: write` (same as Tauri build).
2. Use `npm publish --provenance --access public`.
3. Document in the consuming README how to verify (`npm install --foreground-scripts ... && npm audit signatures`).

No ProtoPulse package is published today. This part is contingent.

## Rollback / revocation

GitHub attestations are append-only (Sigstore transparency log). Revocation is via publication of a corrected attestation; consumers verifying with `gh attestation verify` see all attestations and can detect divergence.

For supply-chain incidents (compromised CI), rotate the compromised tag's GitHub release + publish a new attestation for the corrected artifact.
