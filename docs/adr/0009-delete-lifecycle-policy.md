# ADR 0009: Global Delete Lifecycle Policy (Soft vs Hard Deletes)

## Status
Accepted

## Context
ProtoPulse contains 27+ database tables across multiple domains (Projects, Architecture, Circuit, BOM, Validation, Chat, History, etc.). Historically, the application lacked a consistent deletion strategy. Some tables implemented soft deletes (using a `deletedAt` timestamp) to prevent accidental data loss, while others relied on hard SQL deletes. This inconsistency led to bugs (e.g., orphaned references, missing `isNull(deletedAt)` checks) and made it difficult to reason about data retention and cascade behaviors.

As the platform scales to support team collaboration and complex ECO (Engineering Change Order) workflows, we need a unified policy dictating when data is permanently destroyed versus when it is merely tombstoned.

## Decision
We are adopting a stratified delete lifecycle policy based on entity classification. 

### 1. The Policy Matrix

| Entity Classification | Delete Strategy | Rationale | Examples |
|-----------------------|-----------------|-----------|----------|
| **Core Domain Entities** | **Soft Delete** (`deletedAt`) | High value, user-authored data where accidental deletion causes significant rework. Preserves history for undo/restore and auditing. | `projects`, `architecture_nodes`, `architecture_edges`, `bom_items` |
| **Volatile / Derived State** | **Hard Delete** | High-volume data that is generated by the system (e.g., AI chat, validation checks, simulations) or represents temporary state. Hard deleting prevents unbounded database growth. | `validation_issues`, `chat_messages`, `ai_actions`, `simulation_results`, `history_items` |
| **Deep Relational Graphs** | **Hard Delete** (Cascade) | Deeply nested connectivity data (wires, nets, pins) where soft-delete cascades would be computationally expensive and state-reconciliation in the UI would become unmanageable. | `circuit_designs`, `circuit_instances`, `circuit_nets`, `circuit_wires`, `circuit_vias` |
| **Account / Auth** | **Hard Delete** | PII and security tokens must be destroyed immediately for compliance (GDPR/CCPA) and security hygiene. | `sessions`, `api_keys` |

### 2. Implementation Rules
1. **Query Enforcement:** For any table using Soft Deletes, all `SELECT`, `UPDATE`, and `DELETE` (logical) operations in the storage layer MUST include a `isNull(table.deletedAt)` where clause to prevent leaking tombstoned records into the application state.
2. **Cascading Hard Deletes:** Tables configured for Hard Deletes should rely on PostgreSQL `ON DELETE CASCADE` foreign keys where appropriate to ensure database referential integrity without requiring manual multi-table cleanup in the Express routes.
3. **Restoration:** Soft-deleted entities can only be restored via specific Admin or "Trash Bin" endpoints, preventing regular API flows from accidentally resurrecting them.

## Consequences
- **Positive:** Clear developer expectations. Reduced risk of catastrophic data loss for primary user work (projects, architecture). Automatic pruning of high-volume simulation/validation noise.
- **Negative:** Queries against core tables (Projects, Architecture) must strictly remember to filter by `deletedAt`. If a developer forgets, deleted nodes will magically reappear on the canvas. (Mitigated by centralizing all queries in `server/storage.ts`).
