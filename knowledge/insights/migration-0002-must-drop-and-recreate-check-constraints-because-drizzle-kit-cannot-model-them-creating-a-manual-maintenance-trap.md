# Migration 0002 must drop and recreate CHECK constraints because Drizzle Kit cannot model them

The migration history reveals a non-obvious maintenance trap in the Drizzle ORM workflow:

1. **Migration 0000** (`0000_green_prodigy.sql`): Drizzle Kit auto-generates 19 tables, 15 foreign keys, and 27 indexes — all pure DDL. No CHECK constraints exist because Drizzle schema DSL has no way to express them.

2. **Migration 0001** (`0001_add_enum_constraints.sql`): Hand-written SQL adds 3 CHECK constraints to enforce enum values that Zod validates at the app layer:
   - `chk_chat_role` → `('user', 'assistant', 'system')`
   - `chk_validation_severity` → `('error', 'warning', 'info')`
   - `chk_bom_status` → `('In Stock', 'Low Stock', 'Out of Stock', 'On Order')`

3. **Migration 0002** (`0002_solid_toxin.sql`): Drizzle Kit auto-generates new tables + ALTER columns. Because it sees existing constraints it didn't create, it **drops all three CHECK constraints** (lines 101-103) as part of its diff. Then at the bottom (lines 150-158), hand-written SQL **re-adds the same constraints** with a comment explaining: *"Re-add CHECK constraints from 0001 (dropped above because Drizzle schema does not model them; they are maintained as manual DB-level guards)."*

**The trap:** Every future migration generated by `drizzle-kit generate` that touches these tables will include `DROP CONSTRAINT` statements for the hand-maintained CHECKs. The developer must remember to append re-creation SQL, or the database silently loses its enum enforcement layer, leaving only Zod (which can be bypassed via direct SQL or admin tools).

This is a structural tension between two validation layers:
- **Drizzle/Zod (app layer)**: Validates on every API write, but only protects against application-layer writes
- **CHECK constraints (DB layer)**: Validates on every INSERT/UPDATE regardless of source, but invisible to the ORM

The `_journal.json` shows this is Drizzle migration format version 7 with `breakpoints: true`, confirming the `-->statement-breakpoint` markers. The migration journal tracks only auto-generated metadata (idx, version, when, tag) with no indication of which migrations contain manual SQL additions.

**Connection:** This is related to [[the-schema-insert-pattern-uses-omit-plus-extend-to-create-a-strict-write-contract-while-the-select-type-remains-permissive]] — the Zod insert schemas enforce the same enum values as the CHECK constraints, creating intentional double-validation. The risk is that one layer drifts if the other changes.
