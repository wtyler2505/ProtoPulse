# a11y

Accessibility knowledge for ProtoPulse. The unusual challenge here is that EDA tools are dense, canvas-centric, and coordinate-driven — the patterns that work for content sites (landmarks, headings, reading order) barely apply. The work is figuring out how to make a breadboard, a PCB layer stack, a DRC result surface, and a BOM all navigable by keyboard-only and screen-reader users without collapsing into `role="application"` defeatism.

## Core Principles

- **Multi-channel encoding**: any semantic state must carry meaning on ≥2 of {hue, icon/shape, text label, pattern, position}. See [[multi-channel-severity-encoding-is-the-standard-pattern-for-a11y-compliant-status-ui]].
- **Keep browse mode alive**: `role="application"` is a last resort, not a convenience. See [[role-application-suppresses-screen-reader-browse-mode-and-should-be-avoided-for-mixed-content]].
- **Zone-scoped ARIA roles**: the canvas is not a monolith; break it into semantic zones and apply roles per-zone. See [[aria-grid-pattern-fits-breadboard-terminal-strips-but-not-the-full-canvas-because-power-rails-lack-row-column-semantics]].
- **Focus is real focus**: prefer roving tabindex over `aria-activedescendant` for grids and custom widgets. See [[roving-tabindex-is-more-reliable-than-aria-activedescendant-for-grid-focus-management]].
