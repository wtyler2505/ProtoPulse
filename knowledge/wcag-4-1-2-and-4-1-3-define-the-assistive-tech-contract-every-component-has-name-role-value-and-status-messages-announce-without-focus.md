# WCAG 4.1.2 and 4.1.3 define the assistive-tech contract every component has name role value and status messages announce without focus

SC 4.1.2 Name, Role, Value (Level A) requires that "for all user interface components (including but not limited to: form elements, links and components generated by scripts), the name and role can be programmatically determined; states, properties, and values that can be set by the user can be programmatically set; and notification of changes to these items is available to user agents, including assistive technologies." This is the foundational ARIA criterion — every interactive thing must have a name (accessible label), a role (button, link, slider, etc.), and a value (current state, selection, text content). The criterion is Level A because without it, assistive technologies cannot announce what a component is or what it does, and the interface is literally invisible to screen readers.

The "name" has three sources of truth in priority order: `aria-labelledby` (references another element's text), `aria-label` (explicit string), or the element's own content/value (for native `<button>`, the text content; for `<input>`, the associated `<label>`). Native HTML elements carry correct roles automatically (`<button>` is `role="button"`, `<a href>` is `role="link"`) — the criterion most commonly fails when teams use `<div onClick>` or `<span>` as clickable targets without `role` or `tabindex`, making elements invisible to keyboard and screen readers simultaneously (the button-type-audit failure this project closed in BL-related work). The "value" requirement covers state: a checkbox must announce checked/unchecked, a slider must announce position, a tab must announce selected/unselected, a treeitem must announce expanded/collapsed. A custom component that renders "X of Y selected" only visually — without exposing the same information via `aria-selected` or an equivalent — fails 4.1.2 even if the visual design is perfect.

SC 4.1.3 Status Messages (Level AA, new in WCAG 2.1) extends 4.1.2 to a specific class of content: messages that convey information about the state of the app without moving focus. Toast notifications, form validation summaries, "saved successfully" confirmations, search-result counts, and loading-completed indicators are all status messages. The criterion requires them to be "programmatically determined through role or properties such that they can be presented to the user by assistive technologies without receiving focus." The mechanisms are `role="status"` (equivalent to `aria-live="polite"`), `role="alert"` (equivalent to `aria-live="assertive"`), or explicit `aria-live` on a live region. Without one of these, status messages render to sighted users but are completely silent for screen reader users — which is exactly the failure mode [[radix-toast-ships-aria-live-off-by-default-which-silently-hides-notifications-from-screen-readers]] documents.

Both criteria have silent-failure modes that automated audits miss. For 4.1.2, axe-core verifies that elements have names and roles, but it cannot verify the name is *correct* — a button labeled `aria-label="Click here"` passes axe but fails the purpose of the criterion because "Click here" conveys no information. Similarly, a Value-of-property mismatch (slider shows 50% visually but `aria-valuenow="75"`) passes axe-core (both attributes are present) but is functionally wrong. For 4.1.3, axe-core verifies that live regions exist structurally but cannot verify that new content actually gets announced — timing issues, effective `aria-live="off"` values, and content inserted before the live region is registered all produce silent failures that pass automated tests.

The test strategy both criteria require is the same: **turn on a screen reader and interact with the page**. For 4.1.2, confirm every interactive element announces a meaningful name, its role, and its current state. For 4.1.3, trigger every status-message surface and confirm the message is announced without requiring focus movement. NVDA on Windows, VoiceOver on macOS, and TalkBack on Android are the three minimum targets — they have different behaviors, and passing on one does not guarantee passing on another. For ProtoPulse, every custom widget (the BreadboardLab canvas cells, schematic-node components, BOM table rows) must be manually screen-reader-tested, not merely axe-tested — and the test cases should live in Playwright with real keyboard navigation as covered by [[playwright-focus-trap-testing-requires-real-tab-sequences-not-jsdom]].

The architectural decision implied by both criteria: **every custom component wrapper in the codebase must expose `aria-label`/`aria-labelledby`, role, and state props with validated defaults**. Relying on consumers to remember to pass the right ARIA attributes is the primary regression path — the wrapper should enforce the contract. This is why Radix primitives are valuable: they pre-wire the name/role/value contract and lift consumer responsibility to "provide the label text" rather than "wire up the entire a11y tree."
