## EXECUTE NOW

**Target: $ARGUMENTS**

Parse flags:
- `--coverage` — vault-wide provenance report (skip per-note display).
- `--unverified` — list notes with `confidence: verified` that lack `provenance[]`. These violate T2 schema; should be empty after validate.
- `--json` — emit JSON instead of human-readable.
- `--min-reliability <tier>` — filter to notes whose provenance meets a minimum reliability bar.

**Execute these steps:**

1. **Resolve target** — if a slug was given, find `knowledge/<slug>.md`. Else (no slug, no flags → require slug).
2. **Read frontmatter** via T2's `parse-frontmatter.py` helper (falls back to inline YAML parse if unavailable).
3. **Render** — for a single note: show provenance block as a table. For `--coverage`: tally across all vault notes. For `--unverified`: emit the audit list.

**Pipeline discipline** — read-only. No writes to knowledge/, no pipeline changes.

**START NOW.** Reference below explains provenance semantics + coverage formulas + integration points.

## Integration points

- **T2 frontmatter** — provenance lives in the v2 schema. Source taxonomy enumerated.
- **T3 backlink index** — coverage report can cross-reference "most-consumed + no provenance" = highest-risk notes.
- **T7 health report** — provenance coverage feeds the weekly health summary.
- **T10 extract quality gate** — new notes MUST include provenance when claiming `confidence: verified`; T10 blocks commits that violate.
- **Hover tooltips (UI)** — `<VaultHoverCard>` renders a small badge based on dominant source kind.

## Source taxonomy semantics

| kind | definition | reliability skew |
|------|-----------|------------------|
| `datasheet` | Manufacturer-published spec document | authoritative |
| `standard` | IEEE, IEC, ISO, W3C, WCAG official document | authoritative |
| `vendor-doc` | Non-datasheet official docs (app notes, programming guides) | authoritative |
| `textbook` | Published engineering textbook | authoritative |
| `paper` | Peer-reviewed academic paper | consensus/authoritative |
| `community` | Forum posts, GitHub issues, Stack Overflow | consensus/single-source |
| `experiment` | Direct in-lab measurement | single-source (until replicated) |
| `ai-suggested` | Generated by AI, awaiting human verification | FLAG — should not be cited as verified |
| `code` | Reference to open-source code (with commit SHA) | single-source/authoritative depending |
| `other` | Everything else | single-source |

`ai-suggested` is the CRITICAL flag: never render a note as "verified" in UI if its only provenance is `ai-suggested`. `/vault-source --unverified` specifically surfaces this.

## Anti-patterns

| Anti-Pattern | Why It Fails | Instead |
|--------------|--------------|---------|
| Treat presence of `provenance[]` as truth | An `ai-suggested` entry is just a hypothesis | Check `source` + `verified` fields |
| Cite without `url` | Broken provenance; nothing to verify | Include URL + page for every entry |
| Set `verified-by: tyler` without Tyler confirming | Fabrication | Reserve the field for real confirmations |
| Skip `reliability` | Downstream UI can't prioritize consensus > single-source | Always grade |
| Use `community` for datasheet-verifiable claims | Weakens the citation unnecessarily | Prefer datasheet when available |
